# /rule-conditions — Rule Condition Evaluator Generator

> **In-Process SDK only** — rule condition evaluators deploy inside the ACS JVM as part of the
> Platform JAR. They extend the folder Rules engine and complement action executers
> (generated by `/actions`).

Generate a custom Alfresco rule condition evaluator from requirements.

## Input

Read `REQUIREMENTS.md` to identify rule condition requirements:

1. Resolve the Platform JAR project's `Root path` from Section 2 (Project Architecture).
   - If Section 2 contains no `Platform JAR` project, stop and explain that `/rule-conditions`
     only applies to the in-process Platform JAR project.

2. Read Section 7 (Behaviour Requirements) sub-section "Rule condition requirements".
   - If no rule condition requirements are present in Section 7, stop and ask the user to run
     `/requirements` first (or provide a description as `$ARGUMENTS`).

3. From Section 2, derive:
   - `{platform-project-root}` — `.` for Platform JAR only mode; `{name}-platform/` for Mixed mode
   - `{module-id}` — the Platform JAR **artifactId** (bare artifact ID, e.g. `my-extension`).
     Read from `<artifactId>` in the platform `pom.xml` or derive as `{platform-artifactId}` from
     Section 2. **Never use the full `module.id` property value as the directory name.**
   - `{java-package}` — the Java package declared in Section 2
   - `{prefix}` — the namespace prefix declared in Section 5 (Content Model Requirements)

4. Derive from rule condition requirements:
   - `{ConditionName}` — PascalCase condition name (e.g. `HasAspect`, `IsActiveDocument`)
   - `{condition-id}` — kebab-case condition identifier, prefixed: `{prefix}-{condition-name}`
     (e.g. `sc-has-aspect`, `sc-is-active-document`)
   - Parameters the condition accepts (name, data type, mandatory flag)
   - Which Alfresco services the evaluator needs (e.g. `NodeService`, `ContentService`)

## Output Files

> **All three artefacts below are required and must be generated together in a single run:**
> the evaluator class, the Spring bean registration, and the unit test.

### 1. Condition Evaluator Class
`{platform-project-root}/src/main/java/{package}/action/condition/{ConditionName}Condition.java`

```java
package {package}.action.condition;

import org.alfresco.repo.action.ParameterDefinitionImpl;
import org.alfresco.repo.action.evaluator.ActionConditionEvaluatorAbstractBase;
import org.alfresco.service.cmr.action.ActionCondition;
import org.alfresco.service.cmr.action.ParameterDefinition;
import org.alfresco.service.cmr.dictionary.DataTypeDefinition;
import org.alfresco.service.cmr.repository.NodeRef;
import org.alfresco.service.cmr.repository.NodeService;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import java.util.List;

public class {ConditionName}Condition extends ActionConditionEvaluatorAbstractBase {

    private static final Logger LOG = LoggerFactory.getLogger({ConditionName}Condition.class);

    public static final String NAME = "{condition-id}";

    // Declare parameter constants for each condition parameter
    public static final String PARAM_EXAMPLE = "example-param";

    private NodeService nodeService;

    @Override
    protected boolean evaluateImpl(ActionCondition actionCondition, NodeRef actionedUponNodeRef) {
        LOG.debug("{ConditionName}Condition: evaluating node {}", actionedUponNodeRef);

        // Guard: if node does not exist, condition cannot be satisfied
        if (!nodeService.exists(actionedUponNodeRef)) {
            return false;
        }

        // Read parameter values
        // String paramValue = (String) actionCondition.getParameterValue(PARAM_EXAMPLE);

        // Condition logic — return true if satisfied, false otherwise
        return false; // replace with actual logic
    }

    @Override
    protected void addParameterDefinitions(List<ParameterDefinition> paramList) {
        // Declare each parameter the condition accepts
        paramList.add(new ParameterDefinitionImpl(
            PARAM_EXAMPLE,
            DataTypeDefinition.TEXT,
            false,                          // mandatory flag
            getParamDisplayLabel(PARAM_EXAMPLE)
        ));
    }

    public void setNodeService(NodeService nodeService) {
        this.nodeService = nodeService;
    }
}
```

Key rules for the evaluator class:
- Always extend `ActionConditionEvaluatorAbstractBase` — never implement `ActionConditionEvaluator`
  directly (the abstract base handles registration and `init()` automatically).
- Override `protected boolean evaluateImpl(ActionCondition, NodeRef)` — this is the evaluation
  entry point, not `evaluate()`.
- The `ActionCondition` parameter carries the values supplied when the rule was configured —
  read them with `actionCondition.getParameterValue(PARAM_NAME)`.
- Declare a `public static final String NAME` constant holding the kebab-case `{condition-id}`.
  The parent bean wiring uses this constant via `init()`.
- Declare a `public static final String PARAM_*` constant for every parameter name.
- Override `addParameterDefinitions()` only when the condition has parameters; omit the override
  if the condition takes no parameters.
- Always check `nodeService.exists(actionedUponNodeRef)` before operating on the node.
- Use setter injection only — no `@Autowired`.

### 2. Spring Bean Registration
Add to `{platform-project-root}/src/main/resources/alfresco/module/{module-id}/context/service-context.xml`:

```xml
<bean id="{prefix}.{conditionName}Condition"
      class="{package}.action.condition.{ConditionName}Condition"
      parent="action-condition-evaluator">
    <property name="nodeService" ref="NodeService"/>
    <!-- additional service references as needed -->
</bean>
```

If `service-context.xml` already exists (created by `/actions` or `/behaviours`): append the
bean definition to it. Do **not** create a second `service-context.xml`.

If `service-context.xml` does not yet exist: create it and add the import to `module-context.xml`:
```xml
<import resource="classpath:alfresco/module/{module-id}/context/service-context.xml"/>
```

`{platform-project-root}/src/test/java/{package}/action/condition/{ConditionName}ConditionTest.java`

import static org.junit.jupiter.api.Assertions.assertTrue;
import static org.mockito.Mockito.when;

@ExtendWith(MockitoExtension.class)
class {ConditionName}ConditionTest {

    @Mock NodeService nodeService;
    @Mock ActionCondition actionCondition;

    private {ConditionName}Condition condition;
    private final NodeRef nodeRef = new NodeRef(StoreRef.STORE_REF_WORKSPACE_SPACESSTORE, "test-node-id");

    @BeforeEach

        condition = new {ConditionName}Condition();

        assertFalse(result);
    }

    @Test
    void evaluateImpl_returnsTrue_whenConditionMet() {
        when(nodeService.exists(nodeRef)).thenReturn(true);
        // stub additional service calls as needed

        boolean result = condition.evaluateImpl(actionCondition, nodeRef);

        assertTrue(result);
    }
}

## Conventions
- `{module-id}` is the Platform JAR **artifactId** — the bare artifact ID. Derive it as
  `{platform-artifactId}` from Section 2 or from `<artifactId>` in the platform `pom.xml`.
  **Never use the full `module.id` property value as the directory name.**
- `{platform-project-root}` is `.` for Platform JAR only mode, or `{name}-platform/` for Mixed mode
- Class location: `src/main/java/{package}/action/condition/` (sibling package to `action/executer/`)
- Condition name constant format: `{prefix}-{condition-name}` (kebab-case, prefix-scoped)
- Bean ID format: `{prefix}.{conditionName}Condition`
- Parent bean: `action-condition-evaluator` (not `action-executer`)
- Conditions and action executers share `service-context.xml` — no separate context file needed
- Never generate rule condition classes inside the Event Handler project

## Share UI Note
Registering the bean makes the condition available programmatically (via `ActionService` and the
Workflow/Rules REST APIs). To expose it in the **Share Rules UI** dropdown, a separate
`share-config-custom.xml` entry is required in a Share JAR project — use `/share-config` for that.
