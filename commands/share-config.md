# /share-config — Share UI Configuration Generator

> **Share JAR only** — deploys into the Share web tier, not into the ACS repository tier.

Generate `share-config-custom.xml` and related Share web-extension resources from requirements.

This command covers **forms**, **document-library aspects**, **DocLib UI actions**, **rule-action parameter UI**, **evaluators**, and **indicators**. Repository action executers themselves are generated by `/actions`; this command wires them into Share.

**Conventions source:** file paths follow *Share UI Model → File Placement* in `AGENTS.md`.
Repository action bean ids and Share wiring parameters follow *Share UI Model → Repository action wiring*
in `AGENTS.md` (`{prefix}.{beanName}`, e.g. `sc.enableWebFlag`).

When DocLib actions or action dialogs are required, the Platform JAR must already contain the
action beans from `/actions`. Read `service-context.xml` from the Platform JAR project and reuse
those bean ids exactly — never invent Share-only action ids.

## Input

Read `REQUIREMENTS.md` and identify Share-tier UI requirements.

Resolve:

- the **Share JAR** project's `Root path` from Section 2 (Project Architecture)
- the **Platform JAR** project's `Root path`, if one exists, so custom type/aspect/property names and **action bean IDs** can be reused from the repository model and `/actions`

If Section 2 contains no `Share JAR` project, stop and explain that `/share-config` only applies to Share-tier addon projects.

If the requirement is clearly for ACA/ADF/custom frontend work instead of Share, stop and explain that `/share-config` is the wrong generator.

### Requirement categories to detect

| Category | Typical requirement signal | Primary artefacts |
|----------|---------------------------|-------------------|
| Forms | edit/create/view metadata in Share | `node-type` / `aspect` `<config>` blocks |
| Aspects in DocLib | add/remove aspects from document library | `DocumentLibrary` `<aspects>` block |
| DocLib UI actions | menu items invoking repo actions or URLs | `DocLibActions` `<config>` block |
| Rule action UI | folder picker or custom parameter UI for folder rules | rule-config web script override + JS component |
| Conditional visibility | show/hide actions or indicators by metadata | Spring evaluator beans + `<evaluator>` refs |
| Indicators | icons in document list columns | `DocumentLibrary` `<indicators>` block |

## Output Files

### 1. Main Share configuration

`{share-project-root}/src/main/resources/META-INF/share-config-custom.xml`

Generate or update a **single** root file. The document root **must** be `<alfresco-config>`; append inner `<config>` blocks for each concern inside it — do not split actions or forms into separate XML files. CI (`tests/validate-tutorials/checks/share-config.sh`) asserts the presence of `<alfresco-config>`.

When creating a new file, start from this skeleton:

```xml
<alfresco-config>
    <!-- <config> blocks from §1a–1e -->
</alfresco-config>
```

When updating an existing file, preserve the `<alfresco-config>` wrapper and insert or merge `<config>` blocks inside it.

Subsections §1a–1e show **partial** `<config>` blocks only — never emit a standalone file containing only those snippets.

#### 1a. Type and aspect forms

**Partial block** — append inside the single `<alfresco-config>` root from §1.

```xml
<config evaluator="node-type" condition="{prefix}:{type}">
    <forms>
        <form>
            <field-visibility>
                <show id="{prefix}:{propOne}"/>
                <show id="{prefix}:{assocName}"/>
            </field-visibility>
            <appearance>
                <set id="general" appearance="bordered-panel" label-id="{prefix}.set.general"/>
                <field id="{prefix}:{propOne}" set="general" label-id="{prefix}.{propOne}"/>
                <field id="{prefix}:{assocName}" set="general" label-id="assoc.{prefix}_{assocName}"/>
            </appearance>
        </form>

        <form id="doclib-simple-metadata">
            <field-visibility>
                <show id="{prefix}:{propOne}"/>
            </field-visibility>
            <appearance>
                <field id="{prefix}:{propOne}" label-id="{prefix}.{propOne}"/>
            </appearance>
        </form>
    </forms>
</config>

<config evaluator="aspect" condition="{prefix}:{aspect}">
    <forms>
        <form>
            <field-visibility>
                <show id="{prefix}:{aspectProp}"/>
            </field-visibility>
            <appearance>
                <set id="aspect" appearance="bordered-panel" label-id="{prefix}.set.aspect"/>
                <field id="{prefix}:{aspectProp}" set="aspect" label-id="{prefix}.{aspectProp}"/>
            </appearance>
        </form>
    </forms>
</config>
```

For associations, use `label-id="assoc.{prefix}_{assocLocalName}"` (dots in the QName become underscores in the key).

#### 1b. Document Library — aspect visibility

When requirements ask to expose aspects in the document library (add/remove from documents):

**Partial block** — append inside the single `<alfresco-config>` root from §1.

```xml
<config evaluator="string-compare" condition="DocumentLibrary">
    <aspects>
        <visible>
            <aspect name="{prefix}:{aspect}" />
        </visible>
        <addable>
            <!-- omit entries to inherit from visible -->
        </addable>
        <removeable>
            <!-- omit entries to inherit from visible -->
        </removeable>
    </aspects>
</config>
```

Merge `<aspects>`, `<indicators>`, and other `DocumentLibrary` children into **one** `<config evaluator="string-compare" condition="DocumentLibrary">` block per file.

#### 1c. Document Library — DocLib UI actions

UI actions (document library and document-details menus) live in a **separate** config block with evaluator `DocLibActions`. This is **not** form configuration and **not** repository Spring wiring.

**Partial block** — append inside the single `<alfresco-config>` root from §1.

```xml
<config evaluator="string-compare" condition="DocLibActions">
    <actions>
        <!-- Link action -->
        <action id="{prefix}-web-site" type="link" label="actions.{prefix}.web-site" icon="{prefix}-website">
            <param name="href">https://example.com</param>
            <param name="target">_blank</param>
        </action>

        <!-- Repo action without parameters (single-click) -->
        <action id="{prefix}-web-enable" type="javascript" label="actions.{prefix}.web-enable" icon="{prefix}-create-website">
            <param name="function">onActionSimpleRepoAction</param>
            <permissions>
                <permission allow="true">Write</permission>
            </permissions>
            <param name="action">{prefix}.enableWebFlag</param>
            <param name="successMessage">message.{prefix}.web-flag.enabled</param>
            <param name="failureMessage">message.{prefix}.web-flag.failure</param>
            <evaluator negate="true">{prefix}.evaluator.doclib.action.isActive</evaluator>
        </action>

        <!-- Repo action with parameters (dialog before invoke) -->
        <action id="{prefix}-transform" type="javascript" label="actions.{prefix}.transform" icon="{prefix}-transform">
            <param name="function">onActionFormDialog</param>
            <param name="itemKind">action</param>
            <param name="itemId">{prefix}.setWebFlag</param>
            <param name="mode">create</param>
            <param name="destination">{node.nodeRef}</param>
            <param name="successMessage">message.{prefix}.transform.success</param>
            <param name="failureMessage">message.{prefix}.transform.failure</param>
        </action>

        <!-- Repo action with parameters and submit-disable -->
        <action id="{prefix}-generate-docs" type="javascript" label="actions.{prefix}.generate-docs" icon="{prefix}-generate">
            <param name="function">onActionFormDialogWithSubmitDisable</param>
            <param name="itemKind">action</param>
            <param name="itemId">{prefix}.generateDocuments</param>
            <param name="mode">create</param>
            <param name="destination">{node.nodeRef}</param>
            <param name="successMessage">message.{prefix}.generate.success</param>
            <param name="failureMessage">message.{prefix}.generate.failure</param>
            <evaluator>{prefix}.evaluator.doclib.action.canGenerate</evaluator>
        </action>
    </actions>

    <actionGroups>
        <actionGroup id="document-browse">
            <action index="510" id="{prefix}-web-enable" />
            <action index="520" id="{prefix}-web-disable" />
        </actionGroup>
        <actionGroup id="document-details">
            <action index="510" id="{prefix}-web-enable" />
            <action index="520" id="{prefix}-web-disable" />
        </actionGroup>
    </actionGroups>
</config>
```

**DocLib action rules:**

| Rule | Detail |
|------|--------|
| Action `id` | Project-prefixed kebab-case (e.g. `{prefix}-web-enable`) — **not** the repository bean id |
| Action display text | Use `label="..."` (message bundle key or literal text) — **never** `label-id` on `<action>` |
| `<param name="action">` | Must match the **Spring bean id** from `/actions` (`{prefix}.{beanName}`, e.g. `sc.enableWebFlag`) |
| `type="javascript"` + `onActionSimpleRepoAction` | For parameterless repo actions only; uses `<param name="action">` |
| `type="javascript"` + `onActionFormDialog` | When the repo action has parameters; uses `<param name="itemId">` equal to the bean id + action form (§1e) |
| `type="javascript"` + `onActionFormDialogWithSubmitDisable` | Same parameters as `onActionFormDialog`; disables submit while the dialog is open |
| `type="link"` | External or internal URL; no repo action |
| `<actionGroups>` | Required — actions do not appear in menus without group membership |
| `index` | Higher number = lower in the menu |
| `<evaluator>` | Optional; references a Spring bean id (see §3). Use `negate="true"` to invert |
| OOTB merge | Do **not** copy built-in actions/groups — Share merges custom entries automatically |

**Single-click UI actions with hardcoded parameters:** Share cannot pass hardcoded parameters through `onActionSimpleRepoAction`. When requirements need one-click enable/disable (or similar), generate **separate parameterless** repository actions in `/actions` (e.g. `{prefix}.enableWebFlag`, `{prefix}.disableWebFlag`) and reference those bean ids here — do not point the UI action at a parameterized action executer.

#### 1d. Document Library — indicators

When requirements ask for list-column icons driven by metadata:

**Partial block** — append inside the single `<alfresco-config>` root from §1.

```xml
<config evaluator="string-compare" condition="DocumentLibrary">
    <!-- merge with <aspects> in the same block when both exist -->
    <indicators>
        <indicator id="{prefix}-website" index="10">
            <evaluator>{prefix}.evaluator.doclib.action.isActive</evaluator>
        </indicator>
    </indicators>
</config>
```

Share resolves the icon from `{indicator-id}-16.png` under `META-INF/resources/components/documentlibrary/indicators/`.

#### 1e. Action parameter forms

When a DocLib action uses `onActionFormDialog` or `onActionFormDialogWithSubmitDisable`, define the
parameter form in a **separate** `<config>` block. The `condition` attribute must equal the
repository action Spring bean id (same value as `<param name="itemId">` in §1c).

**Partial block** — append inside the single `<alfresco-config>` root from §1.

```xml
<config evaluator="string-compare" condition="{prefix}.{actionBeanName}">
    <forms>
        <form>
            <field-visibility>
                <show id="{paramOne}"/>
                <show id="{paramTwo}"/>
            </field-visibility>
            <appearance>
                <field id="{paramOne}" label-id="{prefix}.{actionBeanName}.form.field.{paramOne}"/>
                <field id="{paramTwo}" label-id="{prefix}.{actionBeanName}.form.field.{paramTwo}"/>
            </appearance>
        </form>
    </forms>
</config>
```

- Field `id` values match the action executer parameter names from `/actions` (not QNames unless the action parameter is a property)
- Do **not** use a standalone `<form id="...">` referenced by a custom itemId — Share resolves the form via `condition="{prefix}.{actionBeanName}"`
- Label keys: `{prefix}.{actionBeanName}.form.field.{paramName}` in the Share message bundle (§2)

### 3. Share Slingshot Spring context

`{share-project-root}/src/main/resources/alfresco/web-extension/{artifactId}-slingshot-application-context.xml`

Register message bundles and DocLib evaluators:

```xml
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:schemaLocation="http://www.springframework.org/schema/beans
           http://www.springframework.org/schema/beans/spring-beans.xsd">

    <bean id="{artifactId}_resources"
          class="org.springframework.extensions.surf.util.ResourceBundleBootstrapComponent">
        <property name="resourceBundles">
            <list>
                <value>alfresco.web-extension.messages.{module-id}-share</value>
            </list>
        </property>
    </bean>

    <!-- Value evaluator: show action/indicator when property equals value -->
    <bean id="{prefix}.evaluator.doclib.action.isActive"
          parent="evaluator.doclib.action.value">
        <property name="accessor" value="node.properties.{prefix}:isActive" />
        <property name="comparator">
            <bean class="org.alfresco.web.evaluator.StringEqualsComparator">
                <property name="value" value="true" />
            </bean>
        </property>
    </bean>

    <!-- Aspect evaluator example -->
    <bean id="{prefix}.evaluator.doclib.action.hasWebable"
          parent="evaluator.doclib.action.aspect">
        <property name="aspect" value="{prefix}:webable" />
    </bean>
</beans>
```

**Evaluator guidance:**

- Prefer extending OOTB Share evaluators (`evaluator.doclib.action.value`, `evaluator.doclib.action.aspect`, etc.) declared in `slingshot-documentlibrary-context.xml`
- Bean ids must be project-prefixed and referenced from `<evaluator>` in `share-config-custom.xml`
- Evaluator returns `true` → action/indicator **shown**; use `negate="true"` on the action to invert
- Only generate a custom Java `Evaluator` class when OOTB evaluators and chaining cannot express the rule

`{share-project-root}/src/main/resources/META-INF/resources/components/documentlibrary/indicators/{indicator-id}-16.png`

### 5. Rule action parameter UI (when folder rules need custom editors)

Simple repository actions (no parameters) appear automatically in the Share rule action dropdown once the action bean is deployed — **no Share config required**.

When a rule action takes parameters that need a **folder picker**, **path display**, or other custom editor (e.g. `destination-folder` noderef on `{prefix}.moveReplaced`), generate the three-part override below. Do **not** try to solve this with `DocLibActions` or form config.

#### 5a. Rule config web script override

`{share-project-root}/src/main/resources/alfresco/web-extension/site-webscripts/org/alfresco/components/rules/config/rule-config-action.get.config.xml`

Copy structure from the OOTB Share file and customize:

```xml
<rule-config type="action">
    <component>{JavaScriptNamespace}.RuleConfigActionCustom</component>
    <config-definitions webscript="/api/actiondefinitions">
    <menu>
        <group>
            <item id="select"/>
        </group>
        <group>
            <action name="copy"/>
            <action name="move"/>
            <action name="{prefix}.moveReplaced"/>
        </group>
    </menu>
    <action name="copy">Copy</action>
    <action name="move">Move</action>
    <action name="{prefix}.moveReplaced">{JsHandlerName}</action>
</rule-config>
```

- `<action name="...">` under menu groups uses the **repository action Spring bean id** (e.g. `{prefix}.moveReplaced`)
- The text body of `<action name="{prefix}.{actionBeanName}">{JsHandlerName}</action>` is the **JavaScript handler key** inside the custom component (PascalCase, e.g. `MoveReplaced`)

#### 5b. Rule page script includes

Override both templates to load the custom JS:

- `{share-project-root}/src/main/resources/alfresco/web-extension/site-webscripts/org/alfresco/components/rules/rule-details.get.html.ftl`
- `{share-project-root}/src/main/resources/alfresco/web-extension/site-webscripts/org/alfresco/components/rules/rule-edit.get.html.ftl`

Add inside the existing `<@markup id="js">` block:

```html
<@script type="text/javascript"
         src="${url.context}/res/components/{prefix}/rules/config/rule-config-action-custom.js"
         group="rules_custom"></@script>
```

#### 5c. Custom rule config JavaScript component

`{share-project-root}/src/main/resources/META-INF/resources/components/{prefix}/rules/config/rule-config-action-custom.js`

Pattern (extend `Alfresco.RuleConfigAction`, register customisations for the handler name):

```javascript
if (typeof {JsNamespace} == "undefined" || !{JsNamespace}) {
   var {JsNamespace} = {};
}

{JsNamespace}.RuleConfigActionCustom = function(htmlId) {
   {JsNamespace}.RuleConfigActionCustom.superclass.constructor.call(this, htmlId);
   this.name = "{JsNamespace}.RuleConfigActionCustom";
   Alfresco.util.ComponentManager.reregister(this);
   this.customisations = YAHOO.lang.merge(this.customisations, {JsNamespace}.RuleConfigActionCustom.superclass.customisations);
   this.renderers = YAHOO.lang.merge(this.renderers, {JsNamespace}.RuleConfigActionCustom.superclass.renderers);
   return this;
};

YAHOO.extend({JsNamespace}.RuleConfigActionCustom, Alfresco.RuleConfigAction, {
   customisations: {
      {JsHandlerName}: {
         text: function(configDef, ruleConfig, configEl) {
            this._getParamDef(configDef, "{param-name}")._type = "path";
            return configDef;
         },
         edit: function(configDef, ruleConfig, configEl) {
            this._hideParameters(configDef.parameterDefinitions);
            configDef.parameterDefinitions.splice(0, 0, {
               type: "arca:destination-dialog-button",
               displayLabel: this.msg("label.to"),
               _buttonLabel: this.msg("button.select-folder"),
               _destinationParam: "{param-name}"
            });
            return configDef;
         }
      }
   }
});
```

Use the OOTB **Move** action handler as the template when the parameter is a destination folder noderef. Match `{param-name}` to the action executer parameter from `/actions` (e.g. `destination-folder`).

For rule **conditions** from `/rule-conditions`, Share usually renders parameters automatically — custom rule-config JS is primarily for **actions** with noderef/path parameters.

## Rules

### Rule 1 — Reuse repository model and action names exactly

- Reuse type/aspect/property QNames from `/content-model`
- `<param name="action">` and `<param name="itemId">` in DocLib actions must match the action executer **Spring bean id** from `/actions` (`{prefix}.{beanName}`, e.g. `sc.enableWebFlag`), not the Java class name or DocLib action `id`
- Action parameter form `condition` must equal the same bean id (§1e)
- Rule config `<action name="...">` entries use the same repository bean id
- Never invent parallel Share-only model names or action ids

### Rule 2 — Separate concerns into the correct config evaluator

| Concern | Config evaluator | Wrong pattern |
|---------|------------------|---------------|
| Edit/create forms | `node-type` / `aspect` | Putting actions inside form blocks |
| DocLib menu actions | `DocLibActions` | Registering UI actions in `service-context.xml` |
| DocLib action with `label-id` instead of `label` | `DocLibActions` `<action label="...">` | `label-id` on `<action>` — valid only on form `<field>` / `<set>` |
| Action parameter forms | `string-compare` with `condition="{prefix}.{beanName}"` | `<form id="...">` with a custom itemId unrelated to the bean id |
| Aspect pickers in DocLib | `DocumentLibrary` `<aspects>` | Only form config for aspects |
| List indicators | `DocumentLibrary` `<indicators>` | DocLibActions block |
| Rule parameter UI | rule-config web script + JS | DocLibActions or type/aspect `<form>` blocks |

### Rule 3 — Make form intent explicit

- Always generate a full `<form>` block for the type/aspect when forms are required
- Generate `doclib-simple-metadata` when metadata should appear in document-library inline editing
- For `onActionFormDialog` / `onActionFormDialogWithSubmitDisable` actions, define the parameter form in §1e with `condition` equal to the repository action bean id; field ids match action parameter names from `/actions`

### Rule 4 — Prefer OOTB Share evaluators and JS handlers

- DocLib actions: use `onActionSimpleRepoAction`, `onActionFormDialog`, or `type="link"` before custom JS handlers
- Visibility: extend `evaluator.doclib.action.*` beans before writing Java evaluator classes
- Rule folders: reuse `arca:destination-dialog-button` and the Move-action customisation pattern

### Rule 5 — Do not encode repository business rules only in Share

- Share config is presentation and invocation wiring, not business logic
- Validation and side effects belong in repository actions, behaviours, or REST — not in Share XML/JS alone

### Rule 6 — Merge, do not overwrite blindly

- Append or update relevant `<config>` blocks **inside** `<alfresco-config>` in `share-config-custom.xml`
- Do not duplicate `<config evaluator="string-compare" condition="DocumentLibrary">` blocks — merge children
- Do not duplicate `<config evaluator="string-compare" condition="DocLibActions">` blocks

### Rule 7 — Keep deployment boundaries

- All artefacts stay in the Share JAR project
- Never write Share config under `alfresco/module/...`
- Never register action executers or rule conditions in Share Spring context — those belong in the Platform JAR

## Conventions

- `{share-project-root}` is `.` for Share-only mode, or `{name}-share/` for mixed layouts
- `{module-id}` is the Share JAR **artifactId**
- `share-config-custom.xml` lives under `src/main/resources/META-INF/` (Share JAR classpath root — not under `alfresco/web-extension/`)
- `{artifactId}-slingshot-application-context.xml` lives under `alfresco/web-extension/`
- OOTB web script **overrides** (rule config, rule edit/details) go under `alfresco/web-extension/site-webscripts/`
- New custom Surf web scripts (not OOTB overrides) go under `alfresco/site-webscripts/` — use `/surf` for those
- Static resources (icons, rule JS) go under `META-INF/resources/components/...`
- Label keys: project-prefixed; association keys use `assoc.{prefix}_{localName}`
- DocLib `<action>` display text: `label` attribute (message key or literal) — not `label-id`
- Form field/set labels: `label-id` attribute (message bundle key)

## Cross-command workflow

| Need | Command |
|------|---------|
| Action executer Java + repository Spring bean | `/actions` — **run first** when Share menus, dialogs, or rule UI reference repo actions |
| Rule condition evaluator | `/rule-conditions` |
| Share forms, DocLib actions, rule UI, evaluators | `/share-config` (this command) — **requires** Platform JAR beans from `/actions` |
| Custom Surf pages/components | `/surf` |

When requirements include Share menu items, single-click enable/disable, action dialogs, or
folder-picker rule configuration:

1. Generate repository actions with `/actions` (bean ids `{prefix}.{beanName}`)
2. Run `/share-config` — read `service-context.xml` and wire Share using the exact bean ids

When `/actions` requirements mention Share UI exposure, note in the output that `/share-config`
must follow to wire DocLib menus, dialogs, or rule UI.

## Validation

After generating files, verify:

- `share-config-custom.xml` is well-formed XML and lives at `src/main/resources/META-INF/share-config-custom.xml`
- `share-config-custom.xml` uses `<alfresco-config>` as the document root (required by CI)
- referenced QNames match the repository content model when a Platform JAR exists
- every form `label-id`, DocLib action `label`, and message key exists in the Share message bundle
- DocLib `<action>` elements use `label="..."` — no `<action label-id="...">`
- DocLib `<param name="action">` and `<param name="itemId">` values match existing action bean ids in Platform JAR `service-context.xml` (`{prefix}.{beanName}`)
- action parameter form `condition` attributes match the same bean ids as `<param name="itemId">`
- DocLib actions that should appear in menus are listed in `<actionGroups>`
- `{artifactId}-slingshot-application-context.xml` registers the message bundle and any evaluator beans referenced from XML
- rule-config overrides reference parameter names that match `/actions` parameter definitions
- no Share files were written under `alfresco/module/...`
- no action executer beans were added to Share Spring context
