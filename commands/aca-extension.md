# /aca-extension — ACA/ADW UI Extension Generator

> **Out-of-Process, source drop-in** — an ACA/ADW extension is not a Platform JAR.
> It is a folder of Angular TypeScript source that gets copied into an existing
> Alfresco Content Application (ACA) or Alfresco Digital Workspace (ADW) source
> checkout and compiled as part of that application's build.
>
> Reference implementation: `https://github.com/aborroy/alfresco-content-lake-ui`

## Input

Read `REQUIREMENTS.md` to identify UI extension requirements:

1. Read Section 2 (Project Architecture). If it declares a `Platform JAR` project, the
   extension connects to repository REST APIs or Web Scripts generated by other commands.
   The UI extension itself has no Maven build — record only the backend URL base path.

2. Read Section 7 (Behaviour Requirements) sub-section "ACA/ADW extension requirements".
   If none are present, stop and ask the user to run `/requirements` first (or provide a
   description as `$ARGUMENTS`).

3. Derive from requirements:
   - `{ext-name}` — kebab-case extension identifier (e.g. `ext-contract-review`)
   - `{ExtName}` — PascalCase extension name (e.g. `ExtContractReview`)
   - `{extPrefix}` — dot-notation prefix for plugin IDs (e.g. `ext-contract-review`)
   - `{vendor}` — vendor/organisation name for the plugin manifest
   - `{backendBaseUrl}` — proxy path for the backend service (e.g. `/api/contracts`);
     set to `/alfresco/api/-default-/public/alfresco/versions/1` if the extension only
     calls the standard Alfresco REST API
   - `{extension-points}` — the subset of: `page`, `sidebar`, `contextMenu`, `toolbar`
   - `{PageName}` / `{SidebarName}` / `{ServiceName}` — PascalCase component names

Omit `navbar`, `toolbar`, `contextMenu`, and `sidebar` sections for extension points not
requested. Omit `routes` if no page is requested. Keep only the `actions` entries that

### 2. Module / Providers Function
`{ext-name}/src/{ext-name}.module.ts`

```typescript
import { APP_INITIALIZER, EnvironmentProviders, NgModule, Provider } from '@angular/core';
import { HTTP_INTERCEPTORS } from '@angular/common/http';
import { ExtensionService, provideExtensionConfig } from '@alfresco/adf-extensions';
import { provideEffects } from '@ngrx/effects';

import { {ExtName}Effects } from './lib/store/{ext-name}.effects';
// Import only the components that correspond to requested extension points:
import { {PageName}Component } from './lib/components/{page-name}/{page-name}.component';
import { {SidebarName}Component } from './lib/components/{sidebar-name}/{sidebar-name}.component';

export function register{ExtName}Components(extensions: ExtensionService): () => void {
  return () => {
    extensions.setComponents({
      '{extPrefix}.page':    {PageName}Component,     // if page requested
      '{extPrefix}.sidebar': {SidebarName}Component   // if sidebar requested
    });
    // Add custom evaluators here if needed:
    // extensions.setEvaluators({ ... });
  };
}

export function provide{ExtName}Extension(): (Provider | EnvironmentProviders)[] {
  return [
    provideExtensionConfig(['{ext-name}.plugin.json']),
    provideEffects({ExtName}Effects),
    {
      provide: APP_INITIALIZER,
      useFactory: register{ExtName}Components,
      deps: [ExtensionService],
      multi: true
    }
  ];
}

/** @deprecated Use provide{ExtName}Extension() instead. */
@NgModule({ providers: [...provide{ExtName}Extension()] })
export class {ExtName}Module {}
```

Key rules:
- Export both `provide{ExtName}Extension()` and the deprecated `@NgModule` for compatibility.
- Do **not** import components in `@NgModule` `imports` — they are standalone.
- Register components with `ExtensionService.setComponents()` via `APP_INITIALIZER`.
- Use `provideExtensionConfig(['{ext-name}.plugin.json'])` to load the manifest.
- Register NgRx effects via `provideEffects({ExtName}Effects)`.
- Add an HTTP interceptor provider only if the extension calls non-Alfresco backends
  that require custom authentication headers.

### 4. NgRx Effects
`{ext-name}/src/lib/store/{ext-name}.effects.ts`

```typescript
import { Injectable, inject } from '@angular/core';
import { Router } from '@angular/router';
import { Actions, createEffect, ofType } from '@ngrx/effects';
import { tap } from 'rxjs';
import { {EXT_PREFIX}_OPEN_PAGE, {EXT_PREFIX}_ACTION_ON_FILE } from './{ext-name}.actions';

@Injectable()
export class {ExtName}Effects {
  private actions$ = inject(Actions);
  private router   = inject(Router);

  openPage$ = createEffect(
    () => this.actions$.pipe(
      ofType({EXT_PREFIX}_OPEN_PAGE),
      tap(() => void this.router.navigate(['/{ext-name}']))
    ),
    { dispatch: false }
  );

  actionOnFile$ = createEffect(
    () => this.actions$.pipe(
      ofType({EXT_PREFIX}_ACTION_ON_FILE),
      tap((action: any) => {
        const node = action.payload;
        if (node?.id) {
          void this.router.navigate(['/{ext-name}'], {
            queryParams: { nodeId: node.id, name: node.name }
          });
        }
      })
    ),
    { dispatch: false }
  );
}
```

### 5. HTTP Service
`{ext-name}/src/lib/services/{service-name}.service.ts`

```typescript

import { HttpClient } from '@angular/common/http';
import { AppConfigService } from '@alfresco/adf-core';
import { Observable } from 'rxjs';

@Injectable({ providedIn: 'root' })

  private readonly baseUrl: string;

  constructor(
    private http: HttpClient,
    private appConfig: AppConfigService
  ) {
    // URL read from app.config.json — never hardcode
    this.baseUrl = this.appConfig.get<string>(
      'plugins.{extPrefix}Service.baseUrl',
      '{backendBaseUrl}'
    );
  }

  // Add strongly-typed methods for each backend endpoint.
  // Example:
  // getData(nodeId: string): Observable<MyResponse> {
  //   return this.http.get<MyResponse>(`${this.baseUrl}/data/${nodeId}`);

```

Key rules:
- Always read backend URLs from `AppConfigService` — never hardcode paths.
- Use `plugins.{extPrefix}Service.baseUrl` as the config key.
- Keep methods typed with interfaces from `{ext-name}.models.ts`.

## Integration Patches (apply after generation)

These three changes must be made to the ACA or ADW source checkout. They cannot be generated
in this repository — they patch files inside the host application.

### Patch 1 — Register the extension in `extensions.module.ts`

```typescript
// app/src/app/extensions.module.ts  (ACA)
import { provide{ExtName}Extension } from 'projects/{ext-name}/src/public-api';

export function provideApplicationExtensions() {
  return [
    ...provide{ExtName}Extension(),
    // keep existing extensions
  ];
}
```

### Patch 2 — Add plugin JSON to build assets in `project.json`

```json
{
  "glob": "{ext-name}.plugin.json",
  "input": "projects/{ext-name}/src/assets",
  "output": "./assets/plugins"
}
```

Add to `app.targets.build.options.assets` in `app/project.json`.

### Patch 3 — Add backend URL to `app.config.json`

```json
{
  "plugins": {
    "{extPrefix}Service": {
      "baseUrl": "/api/{ext-name}"
    }
  }
}
```

Add to `app/src/app.config.json` (or `app/src/app.config.json`).

## Conventions
- Extension folder name: `{ext-name}` (kebab-case); placed under `projects/` in ACA or
  `libs/` in ADW (Nx workspace).
- Plugin manifest IDs: `{extPrefix}.{element}` — dot-separated, no spaces.
- Action type constants: `{EXT_PREFIX}_{ACTION_NAME}` — UPPER_SNAKE_CASE.
- All Angular components: `standalone: true` — no `NgModule` declarations.
- Backend URLs: always read from `AppConfigService` using `plugins.{extPrefix}Service.baseUrl`.
- Authentication to Alfresco REST APIs is handled automatically by ADF's built-in HTTP
  interceptor — no custom interceptor needed for calls to `/alfresco/api/...`.
- Add a custom HTTP interceptor only when calling non-Alfresco backends that need a ticket
  forwarded (as in the `RagAuthInterceptor` pattern from the reference project).
- Never hardcode `nodeId`, API paths, or credentials in component or service code.
